Core Concepts — Object map

The compressed essence. Print it, pin it, glance at it.

The graph

          CATALOGUE                         PARTIES
   Product ──has many──> Price       Customer ──has saved──> PaymentMethod
   (what)               (how much/        (who)               (a card, bank...)
                         how often)         │ default method ─┘
                            │               │
   ════════════════════════╪═══════════════╪═════════════════════════
   ONE-TIME:    PaymentIntent ──> Charge ──> Balance txn ──> Payout (your bank)
                (the payment)   (an attempt)
   RECURRING:   Subscription ──> Invoice ──> PaymentIntent ──> Charge
                (Customer+Price,  (per cycle,   (same one-time flow on a timer)
                 on a clock)       auto-billed)

The objects in one line each

  • Product (prod_) — what you sell; name + description, no price.
  • Price (price_) — how much + how often; immutable; one Product → many Prices; recurring or one-time.
  • Customer (cus_) — who you charge; the hub for cards, subscriptions, invoices.
  • PaymentMethod (pm_) — a card/bank/wallet; attach to a Customer to reuse.
  • SetupIntent (seti_) — save & validate a card without charging, for later use.
  • PaymentIntent (pi_) — tracks one payment through its statuses to succeeded.
  • Charge (ch_) — one attempt to move money on a PaymentIntent (can be several).
  • Subscription (sub_) — ties a Customer to recurring Price(s); raises an Invoice each cycle.
  • Invoice (in_) — an itemised bill, paid by a PaymentIntent.
  • Payout (po_) — Stripe moving your balance to your bank.

PaymentIntent statuses

requires_payment_method → requires_confirmation → requires_action
   → processing → succeeded            (side exits: canceled, requires_capture)
Fulfil only on succeeded.

Key facts

  • Everything is an object with a prefixed ID; the Dashboard and API act on the same objects.
  • Test mode and live mode are separate accounts with separate keys and data.
  • Amounts are in the smallest currency unit — 2000 = $20.00.
  • Both money flows end in a PaymentIntent + Charge; recurring is one-time on a timer.
  • Webhooks tell your server when async things happen (payment_intent.succeeded, subscription went past_due).
  • Mid-cycle subscription changes → proration line items on the next Invoice — see 10.01 Prorations.

Which integration option?

No website / one-off?            → Payment Link        (no code)
Standard checkout, want it fast? → Checkout            (hosted page — the default)
Must match your own UI exactly?  → Elements            (your page, Stripe's fields)
Genuinely unusual flow?          → Payment Intents API (build it all)
Recurring billing?               → Billing + Checkout + Customer Portal

All four end in the same objects; they differ in how much Stripe builds for you (tax, discounts, address collection, subscriptions).
