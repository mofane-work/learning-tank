The Stripe object model: a map

Stripe looks like a wall of unfamiliar nouns. Underneath, it's a small graph of objects that all connect. Learn the map first and every later detail has somewhere to hang.

Lesson 0001 ~6 min Orientation

Everything is an object with an ID

The single most useful idea for navigating Stripe: almost everything you touch is an object — a record that Stripe stores and hands back with a unique, human-readable ID. The prefix tells you the type at a glance:

cus_NffrFeUfNV2Hib     a Customer
prod_NWjs8kKbJWmuuc    a Product
price_1MoBy5LkdIwH...  a Price
pi_3MtwBwLkdIwHu7...   a PaymentIntent  (a one-time payment)
ch_3MtwBwLkdIwHu7...   a Charge         (one attempt at that payment)
sub_1MowQVLkdIwHu7...  a Subscription
in_1MtHbELkdIwHu7...   an Invoice

You create, retrieve, and update these objects through the Stripe API, and you can see and edit the very same objects in the Dashboard. The Dashboard is just a friendly window onto the objects your API calls create — not a separate system.

Why this helps. Once you see Stripe as "a few dozen object types that point at each other," learning it becomes learning the relationships, not memorising a giant API. That graph is what this whole topic teaches.

Two parallel worlds: test mode and live mode

Every Stripe account is really two separate accounts: test mode and live mode. Test mode takes fake card numbers, moves no real money, and is where you build and experiment. Live mode is production. They have separate data — a customer you create in test mode does not exist in live mode — and separate API keys.

An API key is the secret that authenticates a request and picks which world it acts on. A request signed with a test key creates test objects; a live key creates real ones. Keeping that straight is the first thing that saves you from confusion.

The map

Here is the whole core model on one screen. It has two halves — a catalogue (what you sell) and the parties (who you charge) — joined by two money flows (one-time and recurring).

          CATALOGUE                         PARTIES
   ┌─────────────────────┐         ┌────────────────────────┐
   │  Product            │         │  Customer              │
   │  (what you sell)    │         │  (who you charge)      │
   │      │ has many     │         │      │ has saved       │
   │      ▼              │         │      ▼                 │
   │  Price              │         │  PaymentMethod         │
   │  (how much/how often)│        │  (a card, a bank...)   │
   └─────────┬───────────┘         └───────────┬────────────┘
             │                                 │
             │   ┌─────────────────────────────┘
             ▼   ▼
   ════════ TWO WAYS MONEY MOVES ════════════════════════════

   ONE-TIME            PaymentIntent ──> Charge ──> payout
   (a single sale)     (tracks a payment) (an attempt) (to your bank)

   RECURRING           Subscription ──> Invoice ──> PaymentIntent ──> Charge
   (Customer + Price)  (per billing cycle, auto-billed)

Read it as plain sentences and it's almost obvious:

  • A Price belongs to a Product; one Product can have many Prices (Lesson 2).
  • A Customer can have saved PaymentMethods (Lesson 3).
  • A one-time sale is a PaymentIntent, which results in a Charge, which eventually settles into a payout to your bank (Lesson 4).
  • A Subscription ties a Customer to a Price and, each billing cycle, raises an Invoice that pays itself via a PaymentIntent (Lesson 5).

Notice the payoff in the bottom-right: both money flows end in the same place — a PaymentIntent and a Charge. Recurring billing is, underneath, the one-time payment machinery on a timer. That's why the same objects keep reappearing.

One Stripe, many "products" on top. The objects above are the foundation. Stripe's bigger features are layers on this base: Billing manages Subscriptions and Invoices, Checkout is a ready-made page that creates these objects for you, Connect routes money between multiple accounts. Learn the objects and the products stop being mysterious.

How a request becomes an object — and a webhook

The everyday rhythm of working with Stripe is a three-beat loop:

  1. You call the API — e.g. "create a PaymentIntent for $20".
  2. Stripe creates or updates an object and returns it, with its id and a status.
  3. Stripe emits an event describing what happened (e.g. payment_intent.succeeded) and can deliver it to your server as a webhook — an HTTP call Stripe makes to you.

Webhooks matter because the interesting moment often happens later and elsewhere — a bank confirms a payment, a subscription renews overnight, a card is declined on the next cycle. The webhook is how your system learns about it. We'll only name webhooks here; choosing how to handle them is part of Lesson 6.
