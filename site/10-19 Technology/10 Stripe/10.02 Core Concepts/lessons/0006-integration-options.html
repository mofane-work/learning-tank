Learning Tank Technology Stripe Core Concepts 0006

Your integration options

The payoff. Every Stripe integration creates the objects you've now learned — the only real choice is how much of the work Stripe does for you. Here's the spectrum, and how to pick.

Lesson 0006 ~8 min Choosing an approach

One spectrum: hand it to Stripe ↔ build it yourself

All the options below end in the same place — a PaymentIntent, a Customer, maybe a Subscription. They differ in who builds the checkout experience and the surrounding features (tax, discounts, address collection). More convenience means less control; more control means more to build and maintain.

LESS effort, LESS control                 MORE effort, MORE control
   │                                                            │
   ▼                                                            ▼
Payment Links ──> Checkout ──> Elements ──> Payment Intents API
 (no code)      (hosted page)  (your page,   (you build
                              Stripe's fields)  everything)

1. Payment Links — no code

A shareable URL (or a button you embed) that opens a Stripe-hosted payment page. Stripe: "Use Payment Links to embed or share a link to a Stripe payment page to accept payments without a website." You create it in the Dashboard, paste the link in an email or social post, done.

Reach for it when: you need to take a payment now and don't have (or don't want to build) a checkout — a freelancer's invoice link, a quick product sale, a donation button.

2. Stripe Checkout — the hosted/embedded page

A prebuilt, Stripe-maintained payment page you send customers to (or embed in your site). You create a Checkout Session in code saying what's being sold; Stripe renders the page, collects the card, handles authentication, and creates the objects. It also does the tedious surrounding work for you: "enable Stripe Tax by setting automatic_tax.enabled = true," plus discounts, shipping, address collection, and subscription creation — features you'd otherwise build yourself.

Reach for it when: you have a website and want a production-grade checkout fast, with tax/discounts/subscriptions handled. Stripe explicitly "recommend[s] the Checkout Sessions API for most integrations." This is the sensible default.

Checkout speaks Products and Prices directly. You hand a Checkout Session the Price IDs from Lesson 2, and it builds the page from them. The object model you learned is the integration's vocabulary — which is exactly why we learned it first.

3. Elements — your page, Stripe's secure fields

Stripe Elements are pre-built, secure UI components (the card field, the payment form) you drop into your own page and style to match your brand. Stripe: "Use Stripe Elements to design a custom page to collect payments." The card data still goes straight to Stripe, so you stay out of PCI scope, but the layout and flow are yours.

Reach for it when: you need the checkout to live inside your own UI and match your design precisely, but still want Stripe to handle the sensitive card input.

4. The Payment Intents API directly — build everything

The lowest level: you create and confirm PaymentIntents yourself and own the entire flow. Stripe is blunt about the trade-off: "The Payment Intents API is a lower-level API that allows you to build everything yourself, including tax calculation, discounts, subscriptions, and currency conversion… you calculate the final amount and manage the entire checkout lifecycle in your own code."

Reach for it when: you have an unusual flow that the higher-level tools can't express, and you accept that "some features… require significant effort to replicate."

Billing: the parallel choice for subscriptions

For recurring revenue, Stripe Billing is the layer that manages Subscriptions and Invoices, retries, and prorations for you. The same spectrum applies: let Checkout create the subscription (easy), or drive the Billing APIs yourself (full control). The Customer Portal is a bonus — a Stripe-hosted page where customers update cards, switch plans, or cancel, with no code from you.

The thread that ties it together: webhooks

Whichever option you pick, the decisive moments happen asynchronously — a payment succeeds, a renewal is charged overnight, a card is declined. Webhooks (Lesson 1) are how your system finds out and reacts: grant access on payment_intent.succeeded, revoke it when a subscription goes past_due. Even a no-code Payment Link can fire webhooks. (Webhooks are a topic of their own — a natural next workspace under 10 Stripe.)

A quick decision guide

No website / one-off?            → Payment Link
Standard checkout, want it fast? → Checkout            (the default)
Must match your own UI exactly?  → Elements
Genuinely unusual flow?          → Payment Intents API
Recurring billing?               → Billing + Checkout, with the Customer Portal

You have a normal website and want a solid checkout quickly, with tax and discounts handled. Which option does Stripe recommend as the default?
