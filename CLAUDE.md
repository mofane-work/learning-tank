# Learning Tank

This repository is a personal learning workspace powered by the `/teach` skill, published to GitHub Pages.

## Site Structure

GitHub Pages is served from the repo root. The root `index.html` is a redirect page pointing to `site/`, which is the real site.

```

├── index.html                          ← Redirect to site/
├── site/
│   ├── index.html                      ← Real landing page (table of contents)
│   ├── assets/
│   │   └── style.css                   ← Shared stylesheet for all lessons
│   ├── 10-19 Technology/               ← Area
│   │   ├── index.html                  ← Lists categories in this area
│   │   └── 10 Stripe/                  ← Category (a container, not a workspace)
│   │       ├── index.html              ← Lists topics in this category
│   │       ├── 10.01 Prorations/       ← Topic — one /teach workspace
```

## Johnny Decimal System

Topics are organized using the [Johnny Decimal](https://johnnydecimal.com) system, whose canonical **AC.ID** notation has three levels — **A**rea, **C**ategory, **ID**. We keep that structure and numbering, and call the ID a **topic**:

1. **Area** (`NN-NN`) — a broad domain, e.g. `10-19 Technology`.
2. **Category** (`NN`) — a product, tool, or subject that groups related learning, e.g. `10 Stripe`. This is a **container**, not a `/teach` workspace.
3. **Topic** (`NN.NN`, the JD "ID") — one tightly-scoped thing to learn within the category, e.g. `10.01 Prorations`. **Each topic is one `/teach` workspace.**

A category holds one or more topic workspaces as siblings (`10.01`, `10.02`, …), so related learning lives together. A category with a single topic is fine and expected.

Inside a topic, the lessons (`0001-…`, `0002-…`) are a natural sequential sub-level beneath the JD ID — exactly what Johnny Decimal recommends when a real hierarchy already exists ("if natural hierarchies exist, use them").

### Index pages (drill-down navigation)

Navigation is a **drill-down tree**: there is an `index.html` at every layer, and each one lists only its direct children. This keeps every page small as the site grows — the root never lists individual lessons.

- `site/index.html` → Areas
- `site/NN-NN Area/index.html` → Categories in that area
- `site/NN-NN Area/NN Category/index.html` → Topics in that category
- `site/NN-NN Area/NN Category/NN.NN Topic/index.html` → topic home: that workspace's lessons + reference (the only index that links content files)

All index pages link the single shared stylesheet, so the relative path depends on depth: `assets/style.css` (root), `../assets/style.css` (area), `../../assets/style.css` (category), `../../../assets/style.css` (topic). The whole tree is generated by the `/index` skill — never hand-edit an index page.

### Assigning a new topic

Use the `/new-topic` skill. It will:
1. Determine the correct area for the topic.
2. Choose the category — reuse an existing `NN Category` in that area, or create a new one at the next unused category number.
3. Assign the next unused `NN.NN` topic number within that category.
4. Create `site/NN-NN Area/NN Category/NN.NN Topic/` (e.g. `site/10-19 Technology/10 Stripe/10.01 Prorations/`).
5. Scaffold the workspace ready for `/teach`.

Never manually assign a JD number — always use `/new-topic` to avoid clashes.

### Directory name formats

- **Category:** `NN Category Name/` — two-digit number, a space, then the name in title case. Example: `10 Stripe`, `11 TypeScript`.
- **Topic:** `NN.NN Topic Name/` — the category number, a dot, a two-digit topic number, a space, then the name in title case. Example: `10.01 Prorations`, `10.02 Webhooks`.

## GitHub Pages

- Source: `main` branch, root `/`
- `index.html` at root uses `<meta http-equiv="refresh">` to redirect to `site/`
- All lesson and reference HTML files are accessible at their relative paths under `site/`

## Rules

1. **Always rebuild the index tree after adding or modifying any lesson.** Use the `/index` skill — it regenerates the `index.html` at every layer (root, area, category, topic), not just the root page. The index is the reader's map — never leave it stale, and never hand-edit an index page.
2. **`site/assets/` is shared.** Never duplicate a stylesheet or widget across topics. Put it in `site/assets/` and link to it.
3. **Lesson and reference files both link to `../../../../assets/style.css`** (four levels up from `site/NN-NN Area/NN Category/NN.NN Topic/lessons/` or `.../reference/` — both are the same depth).
4. **One topic per workspace.** If the mission shifts into a genuinely different topic, create a sibling workspace under the same category with `/new-topic` (or a new category if it belongs elsewhere).
5. **Commit after each lesson.** Each lesson is a meaningful unit and should land in its own git commit.
6. **Do not put workspace files (MISSION.md, NOTES.md, etc.) anywhere other than their topic workspace directory.** Keep non-HTML files out of `site/assets/`.
